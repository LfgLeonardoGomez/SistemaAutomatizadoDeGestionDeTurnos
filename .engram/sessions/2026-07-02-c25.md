## Lo que se hizo

### Orquestador (`n8n-workflows/orquestador.json`)

`Normalizar Comando` (Code node) completamente reescrito. Ahora:

1. Parsea callbacks de botones inline: `cmd:crear`, `cmd:crear:rango:*`, `cmd:crear:slot:*`, `cmd:crear:fecha:*`
2. Detecta `cmd:cancelar` y `cmd:reprogramar` (incluyendo `cmd:cancelar:turno_id:*` y `cmd:reprogramar:turno_id:*`)
3. Detecta input de fecha en formato `DD/MM/AAAA` → `payload.fecha_raw`
4. Detecta input de hora en formato `HH:MM` → `payload.slot`
5. Detecta input de número (1-20) → `payload.slot_numero`
6. Detecta comandos de texto: `/reservar`, `/cancelar`, `/reprogramar`, "quiero un turno", "sacar turno", "dar de baja", etc.
7. **Calcula la fecha desde el rango usando timezone Argentina** (Intl.DateTimeFormat con `timeZone: 'America/Argentina/Buenos_Aires'`) y la guarda en **su propio staticData** keyed por chat_id
8. Cuando el usuario clickea un slot o tipea "09:00", lee la fecha de staticData y la pasa en `payload.fecha`

Helpers clave:
- `fmtArgentina(d)`: devuelve "YYYY-MM-DD" en timezone Arg
- `addDays(yyyy_mm_dd, n)`: agrega N días sin problemas de DST

### Sub-flujo (`n8n-workflows/sub-flujo-crear-turno.json`)

Estructura nueva con 5 ramas en el Switch Accion:
- `pedir_rango` → Telegram - Pedir Rango (4 botones)
- `pedir_fecha` → Telegram - Pedir Fecha (input DD/MM/AAAA)
- `fecha_invalida` → Telegram - Fecha Invalida
- `pedir_slot` → HTTP GET → Code - Verificar Slots → Code - Formatear Slots → Telegram - Pedir Slot (texto plano, no botones)
- `crear_turno` → HTTP POST → Code - Extraer Turno ID → Telegram - Turno Creado

El **Code - Verificar Slots** implementa el loop de "+1 día" (7 intentos máx) usando staticData para el contador. Si no hay slots, busca al día siguiente hasta encontrar o rendirse.

El **Telegram - Pedir Slot** ahora muestra texto plano con los slots numerados (no botones inline, porque la UI de n8n para botones dinámicos nos hizo perder mucho tiempo):
```

1. 09:00 - 09:30
2. 09:30 - 10:00
3. 10:30 - 11:00

Respondé con el NÚMERO o la HORA (HH:MM) que querés.
```

## Bug activo: chat_id vacío en Telegram - Pedir Rango

### Síntoma
- El Code - Decidir Paso retorna `accion: "pedir_rango"` correctamente
- El Switch Accion rutea bien a Telegram - Pedir Rango
- Pero el nodo Telegram falla con "chat_id is empty"
- El output del Code - Decidir Paso no tiene el campo `chat_id`

### Causa raíz
**El nodo `Debug - Ver StaticData` que se agregó para diagnóstico estaba en el medio del flujo, entre `Execute Workflow Trigger` y `Code - Decidir Paso`.**

El Debug node tiene este código:
```js
return [{
  json: {
    chat_id_recibido: chatId,    // ← tiene este
    ctx_key_buscado: ctxKey,
    staticData_tiene_ctx: ...,
    // NO tiene "chat_id"
  }
}];
```

El Code - Decidir Paso lee `input.chat_id` del input. Como el input venía del Debug node (no del Execute Workflow Trigger directo), `input.chat_id` era undefined.

### Fix aplicado ✅
**BORRADO el nodo `Debug - Ver StaticData` del sub-flujo en el JSON guardado.** Conexión cambiada: `Execute Workflow Trigger → Code - Decidir Paso` directo.

Si la próxima sesión lo ve en n8n todavía, es porque tiene que reimportar el JSON. El archivo guardado ya está limpio.

## Decisiones de diseño tomadas

| Decisión | Razón |
|----------|-------|
| Botones de rango sí, dropdown de fecha no | Telegram no tiene date picker nativo. El usuario pidió "lo antes posible" para acelerar. |
| Slots como texto numerado, no botones inline | La UI de n8n para botones dinámicos con `inline_keyboard` desde JSON/expression es muy limitada. Texto es más simple y funcional. |
| Input de hora HH:MM, no número de opción | El HH:MM es más natural y no requiere state. |
| Loop +1 día automático, no conversacional | El usuario pidió "sumamos un día y volvemos a pasar fecha exacta" en el chat original. |
| Timezone Argentina explícita en `fmtArgentina()` | El servidor Docker está en UTC. Sin esto, "mañana" salía como "pasado mañana" o un día incorrecto. |
| JWT en lugar de X-API-Key | El endpoint usa `get_current_profesional` que valida JWT, no API key. |
| StaticData en el ORQUESTADOR, no en el sub-flujo | El staticData de n8n es per-workflow. El sub-flujo se reimporta seguido y pierde state. El orquestador es más estable. |
